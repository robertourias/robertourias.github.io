# Spec: ItemCard — componente de card de item no design system

**Status:** draft
**Data:** 2026-05-20
**Autor:** planner-agent

---

## Problema

`apps/tools` e `apps/challenges` exibem listas de itens com cards visuais — ambos com mídia (ícone ou imagem), título, descrição e links de ação — mas cada um implementou o padrão de forma independente e incompatível:

- `apps/tools` usa um `<div>` totalmente customizado fora do design system, sem usar nenhum componente de `@nico.dev/ui`.
- `apps/challenges` tem `ChallengeCard.tsx` construído ad-hoc em cima de `Card` + `CardContent`, com lógica de fallback de imagem misturada na apresentação.

O resultado é inconsistência visual e duplicação de padrão. Toda vez que um novo subprojeto precisar exibir uma lista de itens com esse layout, o padrão se repete fora do design system.

---

## Cenários de Usuário

- **P1 (crítico):** Como desenvolvedor do monorepo, quero um componente `ItemCard` em `@nico.dev/ui` que aceite mídia (ícone ou imagem via slot), título, descrição e links, para não precisar reimplementar esse padrão em cada app.
- **P1 (crítico):** Como visitante de `tools.nico.dev`, quero ver os cards das ferramentas com o mesmo padrão visual dos cards de `challenges.nico.dev`, para ter uma experiência coerente entre os subprojetos.
- **P2 (importante):** Como desenvolvedor, quero ver o `ItemCard` documentado no Storybook com variações de mídia (ícone, imagem, sem mídia), para poder compor o componente corretamente em qualquer contexto.

---

## Requisitos Funcionais

### Design system — `packages/ui` (FR-001–FR-005)

- **FR-001:** Criar `packages/ui/src/components/item-card.tsx` exportando o componente `ItemCard` com a seguinte interface:

  ```ts
  interface ItemCardLink {
    label: string
    href: string
    variant?: "default" | "outline" | "ghost" | "secondary"
    target?: "_blank" | "_self"
    rel?: string
  }

  interface ItemCardProps {
    media?: React.ReactNode        // slot livre: aceita <img>, <Image>, ícone Lucide, emoji, etc.
    title: string
    description?: string | null
    links?: ItemCardLink[]
    className?: string
  }
  ```

- **FR-002:** Layout interno do `ItemCard`:
  - Se `media` for fornecido: área de mídia no topo (sem padding, `overflow-hidden`), seguida de área de texto com padding.
  - Área de texto contém título (`font-semibold text-foreground`), descrição opcional (`text-sm text-muted-foreground line-clamp-2`) e links.
  - Links renderizados como `<Button asChild>` de `@nico.dev/ui`, em linha (`flex gap-2 flex-wrap`), no rodapé da área de texto — separados do conteúdo por `mt-auto`.
  - Sem `href` próprio no card raiz — o app controla se o card inteiro é navegável (via wrapper `<Link>`) ou não.

- **FR-003:** O componente usa apenas tokens do design system (`bg-surface`, `border-border`, `text-foreground`, etc.) — sem valores hardcoded de cor ou espaçamento.

- **FR-004:** Exportar `ItemCard` e `ItemCardLink` no barrel de `packages/ui/src/index.ts`.

- **FR-005:** O componente é um Server Component por padrão (sem `"use client"`). Qualquer lógica de estado (ex: fallback de imagem) fica no app consumidor.

### Storybook — `apps/storybook` (FR-006)

- **FR-006:** Criar `apps/storybook/src/stories/ItemCard.stories.tsx` com as seguintes stories:
  - `ComIcone` — media recebe um ícone Lucide (`<Wrench />`)
  - `ComImagem` — media recebe um `<img>` com `aspect-video object-cover w-full`
  - `SemMidia` — sem prop `media`
  - `SemDescricao` — sem prop `description`
  - `SemLinks` — sem prop `links`
  - `MultiplosBotoes` — dois links (um `default`, um `outline`)

### `apps/tools` — substituição dos cards (FR-007–FR-008)

- **FR-007:** Substituir os `<div>` customizados na home de `apps/tools` (`src/app/page.tsx`) por `ItemCard` de `@nico.dev/ui`:
  - `media`: `<span className="text-3xl">{tool.icon}</span>` (emoji, sem interação)
  - `title`: `tool.name`
  - `description`: `tool.description`
  - `links`: vazio (sem links nos cards de tools — a navegação é feita via wrapper `<Link>` ou via prop futura)
  - Cards com `status === "active"`: envolvidos por `<Link href={tool.href}>` (mesmo comportamento atual).
  - Cards com `status === "coming-soon"`: envolvidos por `<div className="opacity-60 cursor-not-allowed">` com badge "Em breve" abaixo da descrição, conforme comportamento atual. O badge fica no app, não no componente.

- **FR-008:** A home de tools deve manter a mesma grid responsiva: `grid-cols-1 sm:grid-cols-2 lg:grid-cols-3 gap-4`.

### `apps/challenges` — substituição dos cards (FR-009–FR-010)

- **FR-009:** Substituir `apps/challenges/src/components/ChallengeCard.tsx` por uso direto de `ItemCard` de `@nico.dev/ui`:
  - `media`: bloco JSX com `next/image` e fallback via `useState` (lógica de fallback permanece no app, não no componente do design system).
  - `title`: `challenge.name`
  - `description`: `challenge.description`
  - `links`: array com "Ver repositório" (`outline`) e "Ver projeto" (`default`, apenas se `projectUrl` não for `null`).

- **FR-010:** O arquivo `ChallengeCard.tsx` pode ser mantido como wrapper fino que monta o `media` com fallback e repassa para `ItemCard`, ou eliminado e o código movido diretamente para `page.tsx` — a escolha fica a critério do agente de implementação, priorizando o menor número de arquivos.

## Critérios de Sucesso

- [ ] `ItemCard` existe em `packages/ui` e é exportado pelo barrel
- [ ] `pnpm --filter @nico.dev/ui tsc --noEmit` sem erros
- [ ] Storybook exibe as 6 stories sem erros de render
- [ ] `apps/tools` home renderiza todos os cards usando `ItemCard`; cards "active" são clicáveis; "coming-soon" têm badge e opacity
- [ ] `apps/challenges` home renderiza os desafios usando `ItemCard`; imagem com fallback funciona
- [ ] `pnpm --filter @nico.dev/tools build` e `pnpm --filter @nico.dev/challenges build` sem erros
- [ ] Nenhum `<div>` customizado de card remanescente nas homes de tools e challenges

---

## Fora do Escopo

- Estado `disabled` / `coming-soon` dentro do componente `ItemCard` (responsabilidade do app — Opção B escolhida)
- Suporte a `href` no card raiz (sem card clicável no componente — app envolve com `<Link>`)
- Skeleton de loading para o card
- Variantes de layout (horizontal, compacto) — apenas layout vertical nesta iteração
- Migração de outros apps além de `tools` e `challenges`
- Dark mode adicional (já suportado pelos tokens existentes)

---

## Riscos e Premissas

- **Premissa:** `@nico.dev/ui` não depende de `next/image` — o slot `media?: React.ReactNode` resolve isso sem acoplamento ao Next.js.
- **Premissa:** O barrel `packages/ui/src/index.ts` já exporta os componentes existentes; `ItemCard` será adicionado na mesma lista.
- **Risco:** O `CardContent` atual de `@nico.dev/ui` tem `border-t border-border` hardcoded — o `ItemCard` **não** usa `CardContent`; implementa seu próprio layout interno para evitar a borda indesejada quando há mídia.
- **Risco:** A lógica de fallback de imagem em `ChallengeCard` usa `"use client"` + `useState`. Ao mover para wrapper no app, o arquivo de página (`page.tsx`) do challenges deve permanecer Server Component — o wrapper de fallback fica em um componente cliente separado.
